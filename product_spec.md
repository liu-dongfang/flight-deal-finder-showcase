## 1. 项目概述

### 1.1 项目背景
本项目是一个用于美团 AI Coding 考察的网页应用 demo，题目为“特价机票发现平台”。

该项目的评估重点不是复杂工程能力，而是：
- 需求理解是否准确
- 产品设计是否有取舍
- 页面是否具备真实互联网产品气质
- 功能链路是否完整可演示
- 是否体现迭代意识和落地能力

### 1.2 产品定位
这是一个“特价机票发现 + 规则透明化 + 决策辅助”的网页应用，而不是一个真实购票系统。

产品的核心价值不是“帮用户完成下单”，而是：
- 帮用户发现值得关注的低价机票机会
- 帮用户看懂低价背后的隐藏规则和真实成本
- 帮用户判断当前是否值得购买

### 1.3 产品一句话定义
产品名固定为“捡漏机票”。

一句话定义：
一个帮助用户发现高性价比特价机票、快速识别隐藏成本、获得明确购买建议的演示型网页产品。

### 1.4 v1 版本定位
v1 目标是做出一个可本地稳定运行、可完整演示、视觉成熟、链路顺畅的产品原型，不追求真实交易闭环，不追求复杂系统能力。

## 2. 产品目标与非目标

### 2.1 产品目标
- 搭建一个“发现型”而不是“表单型”的机票产品首页
- 提供清晰可理解的结果页，支持“最便宜”和“最划算”两种视图
- 通过卡片信息和详情抽屉解释低价票的隐藏规则
- 用 AI 风格点评提升“智能决策辅助”感
- 用本地 mock 数据完整支撑核心链路与演示稳定性
- 保证页面视觉质量、信息层级、交互流畅度和构建可通过

### 2.2 非目标
- 不构建真实购票流程
- 不做支付
- 不强依赖登录
- 不接入真实第三方航班 API 作为核心能力来源
- 不构建复杂后台、数据库、用户系统
- 不追求功能堆砌，不扩展到“旅行平台”或“OTA 全量能力”

### 2.3 成功标准
如果用户在 2 至 3 分钟演示内能完成以下认知，则说明产品达标：
- 快速感知平台是“发现特价机票机会”的产品
- 能从首页进入结果页并找到有吸引力的航班
- 能区分“看起来便宜”和“真正划算”的差异
- 能在详情中明确理解风险、附加成本与适合人群
- 能感受到产品是一个接近真实互联网产品的 demo，而非课程作业

## 3. 用户问题与设计回应

### 3.1 核心用户问题一：找不到
问题表现：
- 特价信息分散在多个平台和社交内容中
- 用户缺少一个统一入口来发现值得买的低价机票
- 仅靠搜索框无法激发出行灵感，也不利于 demo 展示

设计回应：
- 首页不是空搜索页，而是“搜索 + 发现”双结构
- 首页提供场景化快捷入口，如周末出逃、学生穷游、免签海岛、小长假捡漏
- 首页展示热门特价推荐卡片，让用户一进入就有“机会感”

### 3.2 核心用户问题二：看不懂
问题表现：
- 裸票价低，但不含托运行李
- 退改签限制严格
- 红眼航班、中转时间过短、自助中转、跨航站楼等风险不透明
- 用户看到低价但无法判断是否真的适合自己

设计回应：
- 结果卡片直接展示总价、行李、退改签标签、中转情况
- 详情抽屉集中解释票价构成、规则、风险点
- 所有关键信息优先结构化展示，减少大段文字

### 3.3 核心用户问题三：不知道该不该买
问题表现：
- 用户缺少价格参照系
- 不知道当前价格是低位、常规价还是“假便宜”
- 不知道多花一点是否更值

设计回应：
- 提供“最便宜”和“最划算”两种视图
- 每条结果提供 AI 风格点评，解释其适合人群和主要权衡
- 选择“低价日历”作为唯一主加分功能，为价格判断提供时间维度参照

### 3.4 设计原则
- 发现优先于搜索
- 决策辅助优先于信息堆砌
- 真实总成本优先于裸票价
- 规则透明优先于花哨功能
- 完整链路优先于功能数量
- 产品感优先于技术炫技
- 保持“演示可控”：所有状态应可稳定复现

## 4. v1 范围定义

### 4.1 v1 必做功能

#### A. 首页
- 产品名与一句话卖点
- 搜索区：出发地、目的地、出发日期、返程日期、人数
- 场景化快捷入口
- 热门特价推荐卡片区
- 明确体现“发现平台”属性，不做空白搜索页

#### B. 结果页
- 搜索结果列表
- 顶部筛选区
- 两种视图切换：
  - 最便宜
  - 最划算
- 每张卡片展示：
  - 航司/航班信息
  - 起降时间
  - 飞行时长
  - 中转情况
  - 含税总价
  - 行李信息
  - 退改签标签
  - AI 风格点评

#### C. 航班详情抽屉/弹窗
- 票价构成
- 行李规则
- 退改签说明
- 中转说明
- 风险提醒
- AI 综合建议

#### D. 唯一主加分功能
本项目 v1 只选择一个主加分功能：
- 低价日历

### 4.2 明确不做
- 支付页
- 订单页
- 登录注册
- 真实价格订阅通知
- 多段复杂联程规划
- 机酒打包
- 优惠券体系
- 用户评论社区
- 多语言和多币种
- 复杂地图选点或 3D 航线可视化

### 4.3 为什么选择“低价日历”而不是“收藏/价格追踪”
- 它更直接服务于“现在该不该买”的决策问题
- 它天然强化“发现”和“捡漏”心智
- 它不依赖登录，不引入状态持久化复杂度
- 它更适合 mock 数据稳定演示
- 它能在结果页或详情内形成明显的产品亮点，但不会扩散为复杂系统

## 5. 页面信息架构

### 5.1 页面列表

#### 页面 1：首页 `/`
页面目标：
- 建立产品认知
- 让用户快速进入搜索或发现特价机会

核心模块：
- 顶部导航/品牌区
- 首屏主视觉区与固定卖点说明
- 搜索模块
- 场景快捷入口
- 热门特价推荐区

用户价值：
- 一眼理解产品是做什么的
- 不知道搜什么时也能直接开始浏览
- 快速获得“最近有便宜票”的机会感

#### 页面 2：结果页 `/results`
页面目标：
- 承接搜索行为
- 帮用户在多个候选航班中做比较和判断

核心模块：
- 搜索摘要条
- 筛选区
- 排序/视图切换区
- 结果列表
- 低价日历模块

用户价值：
- 快速缩小范围
- 理解最便宜和最划算的区别
- 同时查看价格、规则、风险、建议

#### 全局交互层：详情抽屉
页面目标：
- 在不跳转离开结果页的前提下解释规则和风险

核心模块：
- 航班基础信息
- 票价构成
- 行李与退改签
- 中转与风险提示
- AI 综合建议

用户价值：
- 避免在多页面跳转中丢失对比上下文
- 提升决策效率

### 5.2 核心跳转关系
- 首页搜索提交 -> 结果页
- 首页场景快捷入口 -> 带预填条件跳转结果页
- 首页热门特价卡片 -> 结果页并预选对应航线/日期
- 结果页点击航班卡片“查看详情” -> 打开详情抽屉
- 结果页点击低价日历某一天 -> 刷新当前结果

### 5.3 主链路
首页进入 -> 通过搜索或快捷入口进入结果页 -> 切换“最便宜/最划算” -> 查看某条航班详情 -> 理解规则与风险 -> 获得购买建议

### 5.4 次链路
- 首页直接点击热门特价卡片 -> 结果页
- 结果页先看低价日历 -> 切换日期 -> 浏览结果
- 结果页反复切换筛选条件 -> 查看不同类型特价

### 5.5 状态设计

#### 空状态
- 首页初始不使用空白状态，而是默认显示推荐内容
- 结果页在无主动搜索时，固定回退到默认落地查询：上海 -> 东京，2026-05-16，单程，1 人，排序为“最划算”，避免“空白结果页”

#### 无结果状态
- 明确提示“当前条件下没有合适特价”
- 提供替代建议：
  - 放宽出发日期
  - 切换附近日期
  - 查看低价日历
  - 返回首页热门推荐

#### 风险状态
- 对红眼、自助中转、跨航站楼、超短衔接、高退改损失使用显著风险标签
- 不阻断浏览，但在卡片和详情中均需可见

## 6. 功能规格（逐页逐模块）

### 6.1 首页

#### 6.1.1 顶部导航/品牌区
模块目标：
- 建立真实产品感和品牌辨识度

建议内容：
- 产品名固定为“捡漏机票”
- 首页固定副标题为“看见低价，也看懂代价”
- 右侧保留轻量辅助入口，不引入登录

用户价值：
- 进入页面即能理解产品定位

#### 6.1.2 首屏主视觉区
模块目标：
- 建立“机会感”和“发现平台”心智

建议内容：
- 主标题固定为“帮你发现真正值得买的特价机票”
- 说明文案固定为“总价透明、规则清晰、帮你判断值不值得买”
- 背景不宜过重，避免喧宾夺主

用户价值：
- 降低理解成本
- 提升点击探索意愿

#### 6.1.3 搜索模块
字段：
- 出发地
- 目的地
- 出发日期
- 返程日期
- 人数

行为：
- 支持默认值与快捷填充
- v1 主演示链路固定为单程
- 首页保留返程日期字段，但它在 v1 中只做轻量展示和 URL 参数保留
- 返程日期不参与结果列表生成、不参与低价日历、不参与排序、不参与筛选、不参与双程价格联动
- 提交后跳转结果页并带 query 参数

用户价值：
- 提供标准机票查询入口

约束：
- 不做复杂校验和真实库存校验
- 只需保证演示状态合理
- 默认落地查询固定为：上海 -> 东京，2026-05-16，单程，1 人

#### 6.1.4 场景快捷入口
建议场景：
- 周末出逃
- 学生穷游
- 免签海岛
- 小长假捡漏

行为：
- 点击后直接预填查询参数并跳转结果页
- 每个场景绑定 2 至 3 条适合展示的 mock 航线

用户价值：
- 用户无需思考搜索词也能开始探索
- 强化“发现平台”属性

#### 6.1.5 热门特价推荐卡片区
卡片字段建议：
- 航线
- 日期范围
- 起价或典型价
- 核心卖点标签
- 一句短评

行为：
- 点击进入对应结果页

用户价值：
- 快速看到“最近值得关注的便宜票”
- 为演示提供更强的内容感

### 6.2 结果页

#### 6.2.1 搜索摘要条
展示内容：
- 出发地 -> 目的地
- 日期
- 人数
- 当前结果数量

行为：
- 支持轻量“修改搜索”入口
- 修改时可返回顶部搜索面板，不单独新开页面

用户价值：
- 保持用户对当前查询上下文的认知

#### 6.2.2 顶部筛选区
v1 筛选项建议限制为：
- 出发时段：清晨 / 白天 / 晚间 / 红眼
- 中转方式：直飞 / 1 次中转 / 不限
- 行李权益：含托运 / 仅随身 / 不限
- 退改签：较灵活 / 严格限制 / 不限
- 风险偏好：隐藏高风险航班 开/关

说明：
- 筛选项不宜过多，避免结果页变成复杂表单
- 所有筛选都必须可用 mock 数据驱动

用户价值：
- 帮助用户快速筛掉明显不合适选项

#### 6.2.2.1 筛选项判定规则表
以下判定全部使用固定映射，不允许后续实现自行改写：

| 筛选项 | 固定取值 | 字段判定规则 |
| --- | --- | --- |
| 出发时段 | 清晨 | 以 `departDateTime` 的本地起飞时间为准，`05:00 <= 起飞时间 < 08:00` |
| 出发时段 | 白天 | 以 `departDateTime` 的本地起飞时间为准，`08:00 <= 起飞时间 < 18:00` |
| 出发时段 | 晚间 | 以 `departDateTime` 的本地起飞时间为准，`18:00 <= 起飞时间 < 23:00` |
| 出发时段 | 红眼 | 以 `departDateTime` 的本地起飞时间为准，`23:00 <= 起飞时间 <= 23:59` 或 `00:00 <= 起飞时间 < 05:00` |
| 中转方式 | 直飞 | `stopCount = 0` |
| 中转方式 | 1 次中转 | `stopCount = 1` |
| 中转方式 | 不限 | 不对 `stopCount` 做过滤 |
| 行李权益 | 含托运 | `checkedBaggage` 不等于 `0kg` |
| 行李权益 | 仅随身 | `checkedBaggage = 0kg` |
| 行李权益 | 不限 | 不对 `checkedBaggage` 做过滤 |
| 退改签 | 较灵活 | `changePolicyLevel = flexible` 或 `changePolicyLevel = limited` |
| 退改签 | 严格限制 | `changePolicyLevel = strict` |
| 退改签 | 不限 | 不对 `changePolicyLevel` 做过滤 |
| 风险偏好 | 隐藏高风险航班 | 过滤满足以下任一条件的航班：`isSelfTransfer = true`、`isCrossTerminal = true`、`stopCount > 0 且 transferMinutes < 90`、`isRedEye = true 且 checkedBaggage = 0kg` |

#### 6.2.3 视图切换：最便宜 / 最划算

最便宜：
- 以含税总价升序排列
- 适合预算优先用户

最划算：
- 以固定公式计算出的 `bestValueScore` 排序
- 评分规则、权重和 tie-breaker 必须严格按 8.2 执行，不允许手工调分
- 页面应明确提示“最划算不是最低价，而是综合体验更优”

用户价值：
- 直接体现产品的决策辅助价值

#### 6.2.4 结果列表
每张航班卡片必须展示：
- 航司与航班号
- 起飞时间 / 到达时间
- 飞行时长
- 直飞或中转说明
- 含税总价
- 行李信息
- 退改签标签
- 风险标签
- AI 风格点评
- 查看详情按钮

卡片信息层级建议：
- 第一层：价格、时间、是否直飞
- 第二层：行李、退改签、风险
- 第三层：AI 点评与适合人群

用户价值：
- 无需打开详情即可完成初筛

#### 6.2.5 航班详情抽屉
打开方式：
- 点击卡片或“查看详情”

信息分区：

1. 航班概览
- 航司、航班号
- 起降机场与时间
- 飞行时长
- 中转链路

2. 票价构成
- 裸票价
- 机建燃油/税费
- 行李附加费用提示
- 预估总成本说明

3. 行李规则
- 随身行李额度
- 托运行李额度
- 超额可能费用提示

4. 退改签说明
- 退票损失
- 改签费用
- 是否支持免费改期窗口

5. 中转说明
- 是否自助中转
- 是否跨航站楼
- 中转停留时长

6. 风险提醒
- 红眼
- 超短衔接
- 自助中转
- 跨航站楼
- 无托运

7. AI 综合建议
- 适合谁
- 为什么推荐或不推荐
- 建议何时下手

用户价值：
- 在最关键的决策环节让“低价规则透明化”

### 6.3 主加分功能：低价日历

#### 6.3.1 功能目标
帮助用户理解当前日期是否值得买，并发现邻近日期更优价格。

#### 6.3.2 展现位置
固定放在结果页列表上方，位于搜索摘要条和结果列表之间。

#### 6.3.3 展示范围
- 固定采用“横向 7 列日期价格条”
- 当前选中出发日期前后各 3 天，共 7 天
- 只展示出发日期价格
- 不做整月网格日历
- 不做返程双日历

#### 6.3.4 显示内容
- 日期
- 当日最低含税价
- 价格等级视觉提示
- 当前选中态

#### 6.3.5 交互
- 点击某一天后刷新结果列表
- 顶部摘要同步更新日期
- 保持已选筛选条件不变

#### 6.3.6 用户价值
- 为“现在买是否合适”提供直观参照
- 强化“捡漏”和“发现机会”的体验

## 7. 数据模型设计（mock data schema）

### 7.1 数据策略原则
- 全部使用本地 mock 数据
- 航线选择贴近真实中国用户出行场景
- 数据要体现差异和可比较性，而不是均质化样本
- 数据字段必须能支撑排序、筛选、AI 点评和详情展示
- 所有状态应稳定复现，不能依赖外部接口

### 7.2 v1 数据规模与数据组织冻结

v1 数据规模固定如下，不允许增减：
- 固定 8 条航线

## 8. 排序与推荐逻辑说明

### 8.1 最便宜排序
规则：
- 按 `totalPrice` 升序
- 若价格相同，按以下固定 tie-breaker 执行：
  - `stopCount` 更少者优先
  - `checkedBaggage` 更高者优先
  - `isRedEye = false` 者优先
  - `flightId` 字典序更小者优先

目标：
- 满足“我只想先看最低价”的直接需求

### 8.2 最划算排序
规则：
- 以 `bestValueScore` 为主排序字段
- `bestValueScore` 必须按以下唯一公式计算，不允许手工调分：

`bestValueScore = 价格分 + 直达分 + 行李分 + 退改签分 + 时段分 - 风险扣分`

固定分值如下：

1. 价格分：基于“当前过滤后结果集”内按 `totalPrice` 升序得到的价格位次
- 第 1 便宜：38 分
- 第 2 便宜：34 分
- 第 3 便宜：30 分
- 第 4 便宜：26 分

2. 直达分
- `stopCount = 0`：20 分
- `stopCount = 1` 且 `isSelfTransfer = false` 且 `transferMinutes >= 120`：10 分
- `stopCount = 1` 且 `isSelfTransfer = false` 且 `90 <= transferMinutes < 120`：6 分
- `stopCount = 1` 且 `isSelfTransfer = true`：0 分
- `stopCount = 1` 且 `isSelfTransfer = false` 且 `transferMinutes < 90`：2 分

3. 行李分
- `checkedBaggage >= 20kg`：15 分
- `checkedBaggage = 15kg`：10 分
- `checkedBaggage = 0kg`：0 分

4. 退改签分
- `changePolicyLevel = flexible`：12 分
- `changePolicyLevel = limited`：6 分
- `changePolicyLevel = strict`：0 分

5. 时段分
- `isRedEye = false`：8 分
- `isRedEye = true`：0 分

6. 风险扣分
- `isSelfTransfer = true`：扣 14 分
- `isCrossTerminal = true`：扣 8 分
- `stopCount > 0` 且 `transferMinutes < 90`：扣 12 分
- `checkedBaggage = 0kg`：扣 8 分

排序 tie-breaker 固定为：
- `bestValueScore` 更高者优先
- 若同分，`totalPrice` 更低者优先
- 若仍相同，`stopCount` 更少者优先
- 若仍相同，`isRedEye = false` 者优先
- 若仍相同，`flightId` 字典序更小者优先

目标：
- 让“略贵但明显更安心”的航班排到前面

冻结约束：
- 在默认查询且不加筛选时，每条航线的“最划算”首位必须是“略贵但综合更优型”
- 在筛选后若仅剩 1 至 2 条结果，允许“最便宜”和“最划算”首位相同；除此之外不允许相同

### 8.3 风险隐藏开关逻辑
当“隐藏高风险航班”开启时，过滤以下任一条件的航班：
- 自助中转
- 跨航站楼
- 超短衔接
- 红眼且无托运

意义：
- 体现产品对普通用户的决策照顾

### 8.4 结果标签逻辑
系统可基于规则自动打以下标签：
- `最低价`
- `性价比优选`
- `规则友好`
- `托运缺失`
- `红眼慎选`
- `中转风险`

这些标签用于：
- 卡片角标
- AI 点评语气辅助
- 详情抽屉摘要提示

## 9. AI 点评生成策略

### 9.1 目标
AI 点评不依赖真实模型推理，但呈现上必须像“智能推荐”，其核心作用是把结构化信息翻译为用户可快速理解的建议。

### 9.2 输出形式

#### 结果页短点评
长度建议：
- 1 至 2 句

表达内容：
- 这张票为什么便宜或为什么划算
- 适合谁
- 最关键的注意事项

示例语气方向：
- “价格压得很低，适合轻装出行，但无托运且退改限制严格。”
- “虽然贵 180 元，但含托运且直飞，整体更适合首次出境用户。”

#### 详情页综合建议
长度建议：
- 2 至 3 句

表达内容：
- 综合评价
- 核心优点
- 核心风险
- 是否建议下手

### 9.3 生成方式
采用规则生成或预设模板填充，不使用真实大模型。

建议规则输入：
- 价格水平
- 是否最低价
- 是否高性价比
- 是否含托运
- 退改签限制
- 中转与风险标签
- 适合人群标签

建议模板结构：
- 结论句：便宜 / 划算 / 有坑 / 慎选
- 依据句：说明价格、行李、退改签、中转条件
- 人群句：适合预算党 / 周末短途 / 带箱用户 / 首次出境用户等

#### 9.3.1 固定模板家族
4 类模板固定如下，不允许新增第 5 类，也不允许改变句式骨架：

| 模板标识 | 结果页短点评固定句式结构 | 详情页综合建议固定句式结构 | 必须出现的关键信息槽位 | 禁止出现的夸张语气 |
| --- | --- | --- | --- | --- |
| `cheap_real` | `价格低于同页多数选项，且{行李信息}，适合{适合人群}。{限制提示}` | `{结论}。它便宜在{价格优势}，同时保留了{核心权益}；需要注意{轻微限制}。` | `价格优势`、`行李信息`、`适合人群`、`1 个限制项` | `史低`、`闭眼冲`、`无脑买`、`神价` |
| `cheap_trap` | `总价最低，但{风险点1}、{风险点2}，只适合{适合人群}。` | `{结论}。便宜点在{低价原因}；代价是{风险点1}、{风险点2}。如需{稳定性诉求}，不建议优先选。` | `最低价身份`、`2 个风险点`、`适合人群`、`反向建议` | `史低`、`闭眼冲`、`无脑买`、`神价` |
| `value_best` | `虽然比最低价贵{价差}元，但{核心优势1}、{核心优势2}，是当前最划算选择。` | `{结论}。多花{价差}元换来{核心权益组合}，整体更适合{适合人群}。若重视稳定性，可优先考虑。` | `价差`、`2 个核心优势`、`适合人群`、`明确推荐` | `史低`、`闭眼冲`、`无脑买`、`神价` |
| `balanced_pick` | `价格和规则比较均衡，{平衡优势}，适合{适合人群}。` | `{结论}。它不是最低价，但在{维度1}和{维度2}之间更平衡；需要留意{轻微限制}。` | `平衡优势`、`适合人群`、`2 个平衡维度`、`1 个轻微限制` | `史低`、`闭眼冲`、`无脑买`、`神价` |

### 9.4 语气要求
- 像智能助手，不像系统报错
- 简洁、可信、具体
- 不要过度营销
- 不使用夸张词汇，如“史低”“闭眼冲”

### 9.5 人群标签建议
- 轻装预算党
- 周末短途用户
- 学生党
- 首次出境用户
- 带箱出行用户
- 时间敏感用户

## 10. 交互说明

### 10.1 整体交互原则
- 关键流程短路径完成
- 信息展示以卡片和结构化标签为主
- 减少跳出式页面，优先原位查看详情
- 动效轻量，服务信息切换，不做炫技

### 10.2 首页交互
- 搜索表单填写后直接进入结果页
- 场景快捷入口支持一键触发
- 热门推荐卡片可点击跳转
- 首屏信息密度适中，避免让用户先滚动很久才能操作

### 10.3 结果页交互
- 顶部保留当前搜索摘要
- 筛选变更后即时刷新列表
- 视图切换后列表排序平滑切换
- 点击卡片或按钮打开详情抽屉
- 点击低价日历日期刷新当前查询结果
- 若存在返程日期，仅在搜索摘要中显示为“返程参考日期”，不参与结果联动

### 10.4 详情抽屉交互
- 从右侧抽屉打开优先于全页跳转
- 支持关闭后保留当前滚动位置和筛选状态
- 风险信息放在首屏可见区域，不埋得过深

### 10.5 关键可演示状态
所有关键控件都必须处于可演示状态，包括：
- 搜索
- 快捷入口
- 筛选
- 排序切换
- 详情打开/关闭
- 低价日历日期切换

### 10.6 需要避免的交互问题
- 大量假按钮不可点击
- 页面切换导致状态丢失
- 详情信息过长需要多次滚动才能看到重点
- 重要标签视觉弱化

## 11. UI 视觉规范

### 11.1 风格定位
- 现代互联网产品风格
- 干净、可信、轻盈
- 强调价格、信息层级和卡片化布局

### 11.2 页面气质要求

#### 首页：机会感
- 首屏应让用户感到“近期有便宜票可捡”
- 推荐卡片应有内容感，但不能像广告墙
- 视觉重点放在搜索与推荐并存

#### 结果页：决策界面感
- 更像比较与判断界面，而不是资讯流
- 价格、标签、风险、点评信息需有明确主次
- 让用户能迅速看出每张卡片之间的差异

#### 详情抽屉：规则解释感
- 以结构化信息块为主
- 重点解释“这张票便宜在哪里，代价是什么”
- 风险提示必须清晰，不可埋入细碎文本

### 11.3 布局建议
- 桌面优先，移动安全
- 首页采用上方首屏主视觉区 + 下方发现内容的分区
- 结果页采用“顶部控制 + 列表主体 + 辅助决策模块”布局
- 卡片设计保持统一节奏和留白

### 11.4 信息层级建议
- 一级信息：价格、时间、直飞/中转
- 二级信息：行李、退改签、风险标签
- 三级信息：AI 点评、适合人群

### 11.5 视觉控件建议
- 价格使用高对比但不刺眼的强调色
- 风险标签采用警示色，但仅用于真正有风险的项
- “最划算”“最低价”等标签需要统一风格
- 日历价格格子需要有明显的低价层级差异

### 11.6 动效建议
- 只保留必要动效：
  - 卡片 hover 轻微反馈
  - 抽屉进出场
  - 视图切换的轻量过渡

必须避免：
- 过度花哨动画
- 炫技型 3D 效果
- 长时间滚动视差
- 与信息判断无关的大面积动态装饰

### 11.7 文案风格
- 简洁直接
- 避免空泛营销词
- 标签优先短词化
- 解释信息优先结构化而非长段落
- 产品名全文统一为“捡漏机票”
- 首页固定副标题全文统一为“看见低价，也看懂代价”
- 页面展示名称、模块标题、按钮文案优先使用中文命名，不保留开放式占位命名

## 12. 技术实现建议

### 12.1 最终推荐技术栈
推荐选择：
- Next.js App Router
- React
- TypeScript
- 本地 mock 数据模块
- 轻量 CSS 方案与组件化实现

### 12.2 技术边界冻结与选择 Next.js 的理由
技术边界固定如下：
- 固定使用 Next.js App Router
- 不使用数据库
- 不使用真实 API Route 作为核心数据来源
- 不使用 Server Actions 作为核心业务依赖
- 首页、结果页、详情抽屉、低价日历的核心数据一律直接从本地 mock 模块读取
- 不通过网络请求拉取核心数据

相较 React + Vite，本项目更推荐 Next.js，理由如下：
- 对 Vercel 部署最友好，默认路径最短
- 自带路由能力，适合首页与结果页这种多页面结构
- 便于后续在不改架构的前提下扩展静态数据加载和 SEO 结构
- 对演示型产品来说，项目结构更清晰，便于组件和页面分层

补充判断：
- 本项目不需要用到 Next.js 的复杂服务端能力
- 可采用前端优先实现方式，以本地数据驱动客户端交互
- 重点使用其路由和部署优势，而不是引入复杂后端逻辑

### 12.3 路由方案建议
- `/`：首页
- `/results`：结果页

说明：
- 航班详情使用抽屉，不新增详情页路由
- 查询参数通过 URL 承载，以保证演示可复现和可分享

建议 URL 参数：
- `from`
- `to`
- `departDate`
- `returnDate`
- `tripType`
- `passengers`
- `sort`
- `scene`

冻结要求：
- `tripType` 默认固定为 `one_way`
- `returnDate` 可存在，但在 v1 中仅用于摘要展示，不驱动数据计算

### 12.4 状态管理方案建议
推荐优先级：
- 页面级状态用 React state
- 查询条件与筛选条件优先同步到 URL search params
- 全局共享轻量状态仅在必要时引入，如当前详情航班、UI 抽屉开关

不建议：
- 一开始引入复杂全局状态库

理由：
- v1 页面少、状态边界清晰
- URL 即状态的一部分，更利于演示和调试
- 不依赖服务端状态，不引入数据库同步问题

### 12.5 mock 数据组织方式
建议本地集中管理并分模块拆分，页面层直接读取：
- 航线数据模块
- 航班数据模块
- 首页推荐数据模块
- 场景数据模块
- 低价日历数据模块

组织原则：
- 数据与 UI 解耦
- 保持字段命名稳定
- 保持固定规模，不在 v1 追加第 9 条航线或额外数据源
- 所有核心数据都必须可通过本地模块静态导入获得

### 12.6 组件分层建议

页面层：
- 首页页面
- 结果页页面

业务模块层：
- 搜索表单
- 场景入口区
- 热门推荐区
- 筛选条
- 排序切换器
- 结果卡片列表
- 航班卡片
- 低价日历
- 详情抽屉

基础组件层：
- 按钮
- 标签
- 卡片容器
- 日期单元
- 信息行
- 风险提示块

原则：
- 先保证业务模块可复用
- 不过度抽象
- 不为“可能的以后”设计复杂组件体系

### 12.7 可测试性建议
后续实现阶段至少覆盖以下可测点：
- 给定查询条件是否能稳定返回预期数据集
- “最便宜”和“最划算”排序是否正确
- 风险过滤是否按预期生效
- 详情抽屉是否能展示完整数据
- 低价日历切换日期后是否映射到正确结果

### 12.8 可部署性建议
- 全部核心数据来自本地 mock
- 避免运行时依赖服务端接口
- 确保生产构建不依赖环境变量才能跑通核心功能
- 构建产物适合直接部署到 Vercel

## 13. 风险与降级方案

### 13.1 风险一：功能扩张导致 demo 漂移
风险：
- 容易从“发现平台”扩展成“购票平台”或“旅行平台”

应对：
- 严格锁定首页、结果页、详情抽屉、低价日历四个核心区域
- 不新增支付、登录、订单、追踪等额外闭环

### 13.2 风险二：数据不真实导致产品感不足
风险：
- mock 数据过于平均，无法支撑“最便宜”和“最划算”的差异

应对：
- 强制每条航线包含“真便宜”“有坑低价”“略贵更优”三类样本
- 强制补齐行李、退改签、中转、风险字段

### 13.3 风险三：信息过多导致结果页变乱
风险：
- 把所有字段都放到卡片首屏会损害判断效率

应对：
- 首屏只保留决策最关键字段
- 复杂规则下沉到详情抽屉

### 13.4 风险四：低价日历实现成本膨胀
风险：
- 若扩展到去返程双日历、复杂月份切换，会偏离 v1

应对：
- v1 固定为结果页列表上方的横向 7 列日期价格条
- 不做返程双日历，不做复杂联动，不做整月价格矩阵

### 13.5 风险五：视觉用力过猛
风险：
- 为了“高级感”加入过多装饰，影响真实产品气质

应对：
- 优先信息层次、留白、字体和卡片秩序
- 动效从简，重点做控件状态和卡片质量

### 13.6 降级方案
若实现阶段时间不足，按以下顺序降级，但不得破坏主链路：

1. 低价日历只允许视觉装饰减法，不允许改变“结果页列表上方横向 7 列日期价格条”的形态
2. 首页推荐卡片数量减少，但保留场景入口和搜索
3. 筛选项从 5 类收缩到 3 类，但保留视图切换和详情抽屉

不得降级的核心：
- 首页发现属性
- 结果页两种视图
- 详情抽屉规则透明化
- AI 风格点评
